# Aisle4 — Obsidian Plugin Specification

## Overview

Aisle4 is a personal Obsidian plugin that reads recipe ingredients from a note
open in the **recipe-view** plugin and sends them to the Aisle4 grocery list API
(`groceries.taiidani.com`). The user reviews and edits the parsed ingredients in
a modal dialog before submission.

Stages 1–3 are complete and working. **Stage 4 (API submission) is not yet
implemented** and is the remaining task described at the end of this document.

### Build & Deploy

`mise run plugin` (run from `/Users/rnixon/taiidani/groceries`) does two things:

1. Bundles `src/main.js` and its imports into `main.js` using **esbuild**
   (installed via `go:github.com/evanw/esbuild/cmd/esbuild` — no npm required).
2. Copies `main.js`, `manifest.json`, and `styles.css` to the vault plugin folder.

esbuild flags: `--bundle --platform=node --format=cjs --external:obsidian`

The `obsidian` module is marked external so it stays as `require('obsidian')` in
the output — Obsidian injects it at runtime. The output format is CommonJS
because Obsidian's plugin loader uses `require()`.

After deploying, reload the plugin in Obsidian via Settings → Community plugins
→ disable/re-enable Aisle4, or Cmd+P → "Reload app without saving".

The `sources`/`outputs` fields in the mise task mean mise skips the run if no
source files have changed since the last bundle.

## Architecture Decisions

### 1. `requestUrl()` instead of `fetch()`

Obsidian runs in Electron. The renderer process enforces CORS like a browser,
and the grocery API has no `Access-Control-Allow-Origin` headers. Native
`fetch()` therefore fails silently with "Failed to fetch". Obsidian's
`requestUrl()` (from the `obsidian` module) routes HTTP through the main Electron
process where CORS is not enforced. **All API calls in this plugin must use
`requestUrl()`**, never `fetch()`.

`requestUrl` API differences from `fetch`:
- Pass `throw: false` to receive non-2xx responses as values rather than throws
- Response body is accessed as `.json` (already parsed), `.text`, `.arrayBuffer`
- Not a Promise-based streaming API; the entire response is buffered

### 2. DOM scraping from recipe-view instead of parsing raw markdown

The plugin requires that the **lachholden/obsidian-recipe-view** plugin
(MIT, https://github.com/lachholden/obsidian-recipe-view) be installed and the
recipe note be open in Recipe View mode. Ingredients are scraped from the live
rendered DOM rather than parsed from the raw markdown file.

Rationale: recipe-view supports quantity scaling (2×, 3× servings). A user
cooking a doubled batch needs doubled quantities on their grocery list. Scraping
the DOM means scaled quantities are automatically captured. Parsing the raw
markdown would always give the unscaled originals.

The recipe-view DOM structure that matters:

- **Two-column layout**: ingredient lists are `<ul>` inside `.column-side`.
  Group sub-headings are `<h3 class="recipe-leaf">` preceding those lists.
- **One-column layout**: ingredient lists are `<ul class="bullets">` (recipe-view
  passes `bullets={true}` only to `CheckableIngredientList` in one-column mode,
  distinguishing them from plain lists in notes/instructions sections).
- **Quantities**: `injectQuantities()` wraps matched numbers+units in
  `<span data-qty="true">` elements before Svelte mounts. The `ScaledQuantity`
  Svelte component inside each span renders the current scaled value.
- **Ingredient text**: `RecipeLeaf.svelte` moves DOM nodes from the original
  rendered `<li>` into a `.leaf > div > .recipe-leaf` subtree on mount. Query
  from `.leaf` to find content.
- **Checked state**: recipe-view's rendered `<li>` elements contain
  `<input type="checkbox">` whose `.checked` property reflects whether the user
  has crossed off that ingredient while cooking.

This coupling is accepted as a known risk. recipe-view is updated very rarely
and the DOM structure is stable.

### 3. Parenthetical alternate-unit quantities

Recipes often annotate ingredients with metric equivalents in parentheses, e.g.
`**1/4 cup (35 g)** thinly sliced garlic`. recipe-view's `injectQuantities()`
wraps both `1/4 cup` and `35 g` in separate `[data-qty]` spans, leaving `(` and
`)` as text node siblings around the second span.

The `isParentheticalQty(span)` helper in `scraper.js` detects this pattern by
checking whether the span's immediate siblings are text nodes ending with `(`
and starting with `)`. Parenthetical spans are excluded from the `quantity` field
and their surrounding `()` are stripped from the `name` field.

**Important**: sibling references must be captured before any span is removed
from the clone, because removing a span changes the sibling chain of adjacent
spans. The code captures `{ span, prev, next, parenthetical }` for all spans
upfront, then processes them.

### 4. Ingredient data shape

`scrapeIngredients()` returns an array of:

```js
{
  group: string | null,  // recipe-view sub-heading (e.g. "Shawarma Blend"), display only
  name: string,          // ingredient name with quantity spans removed
  quantity: string,      // primary quantity as displayed in recipe-view (may be scaled)
  original: string,      // full textContent of .leaf before any processing (for reference)
  doneInRecipeView: bool // true if the checkbox was checked in recipe-view
}
```

`group` is purely for organising the modal display. It does not map to any
primitive in the grocery API and must not be sent to the API.

## The Grocery API

Base URL stored in plugin settings (default: `https://groceries.taiidani.com`).

All protected endpoints require:
```
Authorization: Bearer <token>
```

### Authentication

**`POST /api/v1/auth/login`** — no auth required

Request:
```json
{ "username": "string", "password": "string" }
```

Response (200):
```json
{ "token": "<64-char hex string>", "expires_at": "<RFC3339 timestamp>" }
```

The token is valid for 30 days. It is stored in plugin settings (`settings.token`
and `settings.tokenExpiresAt`). The settings tab has a Connect button that calls
this endpoint and saves the token. The token is displayed with its expiry in
green in the settings tab when valid.

### Adding items to the shopping list

**`POST /api/v1/list/items`** — requires auth

Request (name-based mode — creates the item automatically if unknown):
```json
{ "name": "string", "quantity": "string" }
```

Response (201 Created):
```json
{
  "id": 42,
  "item_id": 1,
  "item_name": "Apples",
  "category_id": 3,
  "quantity": "2 lbs",
  "done": false
}
```

Response (409 Conflict): the item is already on the list. This is a normal
condition (user may have added it earlier) and must be treated as a **soft
warning**, not an error. The response body is:
```json
{ "error": "string" }
```

Error responses generally have the shape `{ "error": "string" }`.

## Completed Stages

### Stage 1 — Plugin scaffold, sidebar icon, settings

- Ribbon icon: `shopping-cart` (Lucide, built into Obsidian — matches the
  grocery webapp's favicon).
- Settings tab: API base URL, username, password, and a Connect button.
- Connect button calls `POST /api/v1/auth/login` via `requestUrl()`, stores the
  token and expiry, and shows the connection status (green/red/muted) in the
  setting row's description area alongside the button.
- Default API base URL: `https://groceries.taiidani.com`

### Stage 2 — Ingredient scraping

- `findRecipeLeaf(app)` in `scraper.js` — finds the recipe-view leaf for the
  currently active file, or returns null.
- `scrapeIngredients(contentEl)` — dispatches to `extractIngredients()` based on
  layout (two-column vs one-column detection).
- `extractIngredients(columnEl, ulSelector)` — walks the DOM in document order,
  tracking `<h3 class="recipe-leaf">` as group labels and extracting `<li>`
  elements as ingredients.
- `isParentheticalQty(span)` — sibling text-node check for alternate-unit
  annotations; see Architecture Decisions above.

If the active note has no recipe-view leaf open, the ribbon click shows:
`"Aisle4: Open this note in Recipe View first."`

### Stage 3 — Review modal

`Aisle4Modal` in `modal.js` takes `(app, ingredients, onSubmit)`.

The modal:
- Is 680px wide (set via `--modal-width` CSS variable on `.aisle4-modal`).
- Shows ingredients grouped by their `group` field under uppercase headings
  separated by a bottom border. Ungrouped recipes (group: null) show no headings.
- Each ingredient is an `.aisle4-ingredient-item` containing:
  - A flex row (`.aisle4-ingredient-row`) with:
    - A checkbox — pre-unchecked if `doneInRecipeView` was true
    - A narrow quantity text input (`.aisle4-qty-input`, 8rem wide)
    - A wide name text input (`.aisle4-name-input`, fills remaining space)
  - A faint hint line (`.aisle4-ingredient-original`) showing the original
    full text from recipe-view, for reference when correcting the parsed fields
- Unchecking a row sets `opacity: 0.35` on the entire `.aisle4-ingredient-item`
  via the `.aisle4-row-excluded` class.
- Footer: Cancel button + "Add to List" CTA button.
- Clicking Cancel closes without calling `onSubmit`.
- Clicking "Add to List" filters `items` to those where `included === true` and
  `name.trim()` is non-empty, calls `onSubmit(selected)`, then closes.

**Current stub in `src/main.js`** (to be replaced in Stage 4):
```js
new Aisle4Modal(this.app, ingredients, selected => {
    console.log('Aisle4 — submitting items:', selected);
    new Notice(`Aisle4: Ready to add ${selected.length} item(s). (API coming in Stage 4)`);
}).open();
```

### What to build

### `src/api.js`

Create and export one function:

```js
import { requestUrl } from 'obsidian';

/**
 * Posts each item in `items` to POST /api/v1/list/items.
 * Returns { added: number, alreadyOnList: number, errors: string[] }.
 * Throws if token is missing.
 */
export async function addToGroceryList(items, settings) { ... }
```

Implementation notes:

- Check `settings.token` is non-empty before starting; throw a descriptive error
  if not (e.g. `"Not connected. Open Settings → Aisle4 and click Connect."`).
- Issue calls **sequentially** (simple `for` loop with `await`). With ~12
  ingredients the latency is negligible and sequential is much easier to reason
  about than `Promise.all` for error handling.
- For each item, call:

```js
await requestUrl({
    url: `${settings.apiBaseUrl}/api/v1/list/items`,
    method: 'POST',
    headers: {
        'Content-Type': 'application/json',
        'Authorization': `Bearer ${settings.token}`,
    },
    body: JSON.stringify({ name: item.name, quantity: item.quantity }),
    throw: false,
})
```

- `201` → increment `added`
- `409` → increment `alreadyOnList` (not an error — item was already on the list)
- Anything else → push `"${item.name}: ${body.error || 'HTTP N'}"` onto `errors`

Return `{ added, alreadyOnList, errors }`.

### Changes to `src/modal.js`

The "Add to List" button handler currently calls `this.onSubmit(selected)`
synchronously. It needs to `await` an async `onSubmit` and show loading state:

```js
addBtn.addEventListener('click', async () => {
    const selected = this.items.filter(it => it.included && it.name.trim());
    if (selected.length === 0) {
        new Notice('Aisle4: No items selected.');
        return;
    }
    addBtn.disabled = true;
    addBtn.textContent = 'Adding…';
    try {
        await this.onSubmit(selected);
    } finally {
        this.close();
    }
});
```

### Changes to `src/main.js`

Import `addToGroceryList` from `./api.js` and replace the stub callback:

```js
import { addToGroceryList } from './api.js';

// inside onRibbonClick():
new Aisle4Modal(this.app, ingredients, async selected => {
    const { added, alreadyOnList, errors } = await addToGroceryList(selected, this.settings);

    const parts = [];
    if (added > 0)          parts.push(`${added} item${added === 1 ? '' : 's'} added`);
    if (alreadyOnList > 0)  parts.push(`${alreadyOnList} already on list`);
    if (errors.length > 0)  parts.push(`${errors.length} failed`);

    new Notice(`Aisle4: ${parts.join(', ')}.`, errors.length > 0 ? 8000 : 4000);

    if (errors.length > 0) {
        console.error('Aisle4 — errors adding items:', errors);
    }
}).open();
```

## Dependency Note

The plugin hard-depends on **lachholden/obsidian-recipe-view** being installed
and enabled in the vault. If the active note is not open in Recipe View mode,
`findRecipeLeaf()` returns null and the ribbon click shows a friendly error.
No attempt is made to parse raw markdown as a fallback.
